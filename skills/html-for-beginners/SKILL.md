# HTML for Beginners

긴 블로그 글·기술 문서·PDF 가이드를 **초보자도 따라올 수 있는 한국어 HTML 학습 자료**로 변환하는 스킬.
압축이 아닌 풀어쓰기, 용어 해설, 일상 비유가 핵심.

## 핵심 원칙

1. **압축보다 풀어쓰기.** 원문이 짧으면 더 풀고, 길면 본질만 살리되 디테일은 보존.
2. **모든 전문 용어는 한 번 풀이.** 영어 약자는 풀어서 표기 + 한국어 의미 + 짧은 일상 비유.
3. **개인 컨텍스트 주입 금지.** userMemories의 회사·프로젝트·스킬 등을 본문에 끌어들이지 않는다. 마지막 "시도해볼 만한 것" 섹션도 누구나 적용 가능한 일반 제안만.
4. **단일 HTML 파일.** 외부 의존성은 CDN 폰트 2종만. JS·외부 CSS 라이브러리 사용 금지.
5. **결과는 `/mnt/user-data/outputs/`에 저장 후 `present_files`로 제시.**

## 워크플로우

### Step 1 — 콘텐츠 수집

입력 형태에 따라 분기:

- **URL이 주어졌으면**: `web_fetch`로 본문 가져오기.
- **PDF 가이드로 연결되면**: 본문에 "Check it out here", "Read the full PDF", PDF 링크 패턴이 보이면 **반드시 PDF까지 따라가서 fetch**. 보통 블로그 본문은 짧고 진짜 내용은 PDF에 있다.
- **텍스트가 직접 주어지면**: 그대로 사용.
- **이미 업로드된 파일**: file-reading 스킬 우선 적용 후 텍스트 추출.

### Step 2 — 구조 설계

다음 섹션 패턴을 따른다. **0번과 1번은 거의 항상 포함**, 마지막 "시도해볼 만한 것"도 필수.

```
kicker (작은 카테고리 라벨)
↓
h1 (제목, 2줄 가능)
↓
sub (한 문장 요약)
↓
─── 본문 시작 ───
0. 먼저, X가 뭔가?       ← 기본 개념·배경
1. 한 줄 요약            ← hero-analogy 박스로 핵심 메시지
2. ~ N. 본문 섹션         ← 빨간 원 번호 h2 + h2-sub 부제
마지막. 시도해볼 만한 것    ← 검정 takeaway 박스
```

원문이 짧으면 본문 섹션이 2~3개, 길면 5~7개 정도. **억지로 늘리지 말 것.**

### Step 3 — 박스 사용 결정

본문 흐름에 따라 4가지 박스를 적재적소에 배치. 자세한 사용법은 `references/design-system.md` 참조.

| 박스 종류 | CSS 클래스 | 언제 쓰는가 |
|---|---|---|
| 용어 박스 (노랑) | `.term` | 영어 약자·전문 용어가 본문에 처음 등장할 때 |
| 비유 박스 (파랑) | `.analogy` | 추상 개념을 일상 사례로 설명할 때 |
| 함정 박스 (빨강) | `.danger` | 흔히 실수하는 패턴·안티패턴을 강조할 때 |
| 해결책 박스 (녹색) | `.good` | 함정에 대응하는 권장 방법을 짝지을 때 |
| 영웅 비유 (큰 빨강) | `.hero-analogy` | 1번 섹션의 한 줄 요약 |
| 일반 카드 (흰색) | `.card-block` | 사용 사례·단계·항목이 여러 개 나열될 때 (`.case-label`로 "사례 N" 라벨) |
| 인용 박스 (회색 이탤릭) | `.prompt-box` | 예시 프롬프트·원문 발췌·명령어 출력을 그대로 분리할 때 |
| takeaway (검정) | `.try` | 마지막 "시도해볼 만한 것" |

빨강·녹색은 짝으로 쓰는 게 효과적 (함정 → 해결). 사용 사례가 3개 이상 나열되는 글(IT 튜토리얼·기능 소개)에서는 `.card-block` + `.case-label`이 주력 박스가 된다.

#### 글 종류별 박스 조합 (참고용 — 규칙 아님)

원문 성격에 따라 어떤 박스가 주력이 되는지 대략의 경향. **이건 체크리스트가 아니라 출발점이다. 원문이 시키는 대로 따르고, 아래 표에 억지로 맞추지 말 것.**

| 원문 성격 | 자주 주력이 되는 박스 | 비고 |
|---|---|---|
| 회사·기관 공지 (announcement) | `.term` 여러 개 + `.hero-analogy` + 함정/해결 1쌍 | 개념 정의가 핵심 |
| IT 튜토리얼·기능 소개 (how-to) | `.card-block`+`.case-label` 사례 묶음 + `.prompt-box` + `.term` | 사례·예시가 본체 |
| 개념 설명·해설 (explainer) | `.term` + `.analogy` 비유 중심 | 비유로 직관 잡기 |

표에 없는 성격의 글이거나 섞여 있으면, 박스 표(위)를 보고 본문 흐름에 맞게 직접 판단한다. **이 표는 2~3개 샘플에서 관찰한 경향일 뿐이라 과신 금지.**

### Step 4 — HTML 생성

`assets/template.html`을 시작점으로 사용. 다음 영역만 채우면 된다:

- `<title>` 태그
- `.kicker`, `<h1>`, `.sub` (헤더)
- 본문 섹션들 (`<h2>` + 각종 박스)
- `<div class="meta">` 출처 표기

**HTML 작성 시 주의:**

- `.kicker`(상단 라벨)와 `.meta`(하단 출처)는 **원문에서 직접 확인한 값만** 적는다. 발행처·날짜·저자를 추측으로 넣지 말 것.
  - 저자명이 원문에 명시돼 있으면 kicker에 `발행처 · 날짜 · 저자` 형식으로 포함. 예: `CLAUDE BLOG · 2026.05.20 · Thariq Shihipar`
  - 회사·기관 공지처럼 개인 저자가 없으면 `발행처 · 날짜`만. 예: `ANTHROPIC · 2026.05.19`
  - 날짜는 원문 표기를 그대로. 요일을 임의로 붙이지 말 것.
- 모든 영어 약자는 첫 등장 시 **풀어쓴 영어 + 한국어 뜻**을 같이. 예: `MCP (Model Context Protocol, 모델 컨텍스트 프로토콜)`
- 본문에서 강조하고 싶은 짧은 용어는 `<em class="t">` (빨강 배지 스타일)
- 본문 prose의 **핵심 문장·구절**은 `<span class="hl">…</span>` (형광 하이라이트, 로드 시 스르륵 자동 재생). 한 글에 2~4개만. `<em class="t">`(짧은 용어)와 역할이 다르다. 색 띤 박스 안에서는 쓰지 말 것 — 본문/흰 카드 위에서만. 호버·스크롤 트리거는 모바일·무JS 제약 때문에 금지(자동 재생만). 자세한 건 `references/design-system.md` 참조.
- 코드 단어는 `<code>` (인라인 회색 배지)
- 표는 가급적 피하고 박스 묶음으로 (모바일에서 깨지기 쉬움)
- 절대로 다음 폰트를 임의로 추가하지 말 것: 템플릿의 Pretendard + Noto Serif KR 외 추가 금지

### Step 5 — 출력 및 제시

```python
# 파일명: claude_<짧은_영어_주제>.html 또는 <짧은_영어_주제>_for_beginners.html
# 위치: /mnt/user-data/outputs/

# 마지막에 반드시:
present_files(["/mnt/user-data/outputs/<filename>.html"])
```

파일 제시 후 한국어로 짧은 요약 코멘트 (2~4줄). 본문에 무엇이 들어갔는지, 어떤 박스를 썼는지 정도. 긴 후기는 불필요.

## 톤 강제 규칙

### 해야 하는 것

- **친근하지만 군더더기 없는 한국어.** 존댓말 기본.
- **영어 용어 + 한국어 풀이 + 비유 3종 세트.** 중요한 용어는 이걸 다 갖춰야 한다.
- **숫자·예시로 감 잡기.** "4K 모니터의 16픽셀 체크박스가 720p로 줄면 5픽셀" 같은 구체.
- **"왜 이게 어려운가"부터.** 해결책 전에 문제부터 풀어쓰기.

### 하지 말아야 하는 것

- **압축**. "한 줄 요약하면" 식으로 본문을 줄이지 말 것. 1번 섹션 hero-analogy 외엔 풀어쓰기.
- **userMemories의 회사·프로젝트·도구 인용**. 본문에도, 마지막 takeaway에도 금지.
- **"인범님 컨텍스트로는…"** 같은 개인화 단락. "일반 사용자도 가져갈 만한 것"만 허용.
- **이모지 남발**. 거의 안 쓰는 게 원칙. 정 필요하면 섹션 헤더에 한 개 이하.
- **표(<table>) 남용**. 모바일 깨짐. 박스 카드로 대체.
- **외부 JS 라이브러리, 아이콘 폰트, Tailwind CDN**. 전부 금지.

## 길이 가이드

원문 분량별 권장 출력 길이:

| 원문 분량 | 출력 본문 섹션 수 | HTML 라인 수 |
|---|---|---|
| 짧은 블로그 (~5분 읽기) | 3~4개 | ~400~500 |
| 중간 길이 + 짧은 PDF | 5~6개 | ~600~800 |
| 긴 PDF 가이드 (~30페이지) | 6~8개 | ~800~1200 |

라인 수가 1500을 넘으면 너무 길다는 신호 — 무엇을 잘라낼지 다시 검토.

**예외 — 사례형 글:** 사용 사례·예시가 많은 글(IT 튜토리얼, 기능 소개)은 사례 카드 하나하나가 부피를 차지해 1300~1500라인까지 자연스럽게 늘어날 수 있다. 이때는 라인 수보다 **"사례를 억지로 늘리거나 줄이지 않았는가"**가 기준. 원문에 사례 5개가 있으면 5개를 다 살리되, 각 카드는 (제목 + 2~3문장 설명 + 인용 1개) 정도로 균일하게 유지한다.

## 흔한 실수와 대응

**실수 1 — 박스 남용**
모든 단락에 박스를 두르면 본문이 사라진다. 박스는 강조용. **prose 70% + 박스 30%** 정도가 적정.

**실수 2 — 영어 용어 풀이 누락**
"MCP를 사용하여..."처럼 풀이 없이 약자만 쓰는 것. 첫 등장 시 무조건 풀이 박스.

**실수 3 — 압축 욕구**
"이걸 한 마디로 정리하면…" 식으로 줄이려는 본능. **이 스킬의 목표는 정반대 — 풀어쓰기.** 1번 hero 외엔 짧게 쓰지 말 것.

**실수 4 — 마지막 takeaway에 개인화 침투**
"인범님의 EPC 도메인에서는…" 식 금지. 누구든 적용할 수 있는 일반 행동 제안만.

**실수 5 — 폰트·디자인 토큰 임의 변경**
Pretendard + Noto Serif KR + `#e63946` 액센트는 시리즈 일관성을 위한 고정값. 절대 바꾸지 말 것.

**실수 6 — 인용 박스로 원문 재현**
`.prompt-box`는 짧은 발췌·예시용. 원문 문단을 통째로 옮겨 담으면 저작권 문제 + 풀어쓰기라는 스킬 목표에도 어긋난다. 인용은 한 문장 수준으로, 나머지는 본인 말로 풀어쓸 것.
