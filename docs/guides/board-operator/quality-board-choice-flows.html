Board operator guide

Quality Board 선택지별 상세 흐름 가이드

Queue에서 어떤 버튼을 눌러야 하는지, 버튼을 누른 뒤 무엇이 끝났고 무엇이 남는지, 그리고 Anchors, Evaluators, Reports가 Queue 이후 어디에 붙는지 한 번에 판단하기 위한 문서입니다.

Live Quality Board 열기 전체 탭 루프 보기 전체 순서도 보기 가장 빠른 판단표로 이동 tech-scout 예시 보기

0. Quality Board 전체 탭 루프

Quality Board는 Queue만 보는 화면이 아닙니다. Queue에서 사람 판정을 끝내고, 필요한 경우 그 판정을 anchor로 저장한 뒤, evaluator 후보를 replay하고, 마지막으로 report로 현재 상태를 남기는 루프입니다.

1 Queue 사람이 review item을 보고 verdict를 기록합니다. 여기서 품질 판단 자체가 끝나거나 rework/evidence 흐름이 열립니다.
2 Save training case 반복 가능한 verdict만 evaluator가 배울 사례로 저장합니다. 저장하지 않아도 Queue verdict는 유지됩니다.
3 Anchors 저장된 human precedent를 봅니다. 이 anchor가 evaluator candidate에 반영됐는지 확인합니다.
4 Evaluators anchor에서 생긴 candidate evaluator를 replay합니다. replay가 passed여야 production으로 올릴 수 있습니다.
5 Reports 오늘의 queue, evidence, anchor, evaluator 상태를 dated report로 저장합니다. 변경 후에는 다시 생성합니다.
탭 언제 보는가 주요 입력 저장/실행 후 생기는 것 끝났다고 보는 기준
Queue

사람 판정이 필요한 review item이 있을 때 봅니다.

매일 우선 확인

가장 앞단입니다. 여기서 verdict가 없으면 나머지 탭으로 배울 재료도 없습니다.

Verdict note

Pass, Request changes, Needs evidence, Fail, Dismiss 중 하나와 이유를 기록합니다.

상태 변경 또는 issue 생성

verdict에 따라 resolved_pass, changes_requested, evidence_collecting 등이 됩니다.

Human review complete

화면에 recorded verdict가 보이면 사람 판정은 끝난 것입니다.

Anchors

저장된 학습 사례가 evaluator에 반영됐는지 확인할 때 봅니다.

Queue verdict 뒤

Save training case를 누른 뒤 생성됩니다.

Case title

사람이 수정한 reusable lesson 제목이 anchor의 핵심입니다.

Candidate evaluator 생성

anchor 저장 시 evaluator candidate version과 queued replay run이 함께 만들어집니다.

Reflected in evaluator: yes

source anchor가 evaluator version에 연결되어 있으면 반영된 상태입니다.

Evaluators

자동 평가기를 실제 운영에 올릴지 검증할 때 봅니다.

Anchor가 있을 때

candidate version은 anchor에서 만들어집니다.

Replay

과거 anchor case에 evaluator 후보를 적용해 같은 판단을 내는지 봅니다.

passed 또는 failed replay

regression이 없으면 passed, 있으면 failed입니다.

Production promotion

passed replay가 있어야 production evaluator로 승격할 수 있습니다.

Reports

현재 Quality Board 상태를 날짜별 스냅샷으로 남길 때 봅니다.

판정이나 증거 변경 후

Queue 상태가 바뀌면 기존 report가 stale할 수 있습니다.

Generate today's report

오늘 날짜의 report를 생성하거나 같은 날짜 report를 갱신합니다.

summary 저장

pending review, needs evidence, anchor gap, evidence stats가 기록됩니다.

최신 report

review item이 report 이후 업데이트되지 않았고, 상태가 현재 board를 설명하면 됩니다.

1. Queue 선택지 판단 순서도

먼저 이 순서도만 따라갑니다. 아래 상세 설명은 각 경로를 더 자세히 풀어쓴 것입니다.

1 Queue 카드가 들어온다

Final QA, Plan QA, Delivery QA, 수동 등록 중 하나가 review item을 만듭니다.

2 카드를 같은 순서로 읽는다

Decision focus → Recommended action → Raw QA excerpt → Evidence → Mission context 순서로 봅니다.

3 하나의 경로만 고른다

판단 가능 여부와 결함 성격에 따라 아래 다섯 경로 중 하나로 내려갑니다.

보이는 상황 누를 버튼 저장 직후 상태 시스템이 여는 다음 일 사람이 다음에 볼 것
판단 대상이 아님

중복, 오래됨, smoke/test, 또는 현재 산출물 판단과 무관합니다.

Dismiss

왜 제외하는지 한 줄로 씁니다.

dismissed

활성 Queue에서 제외됩니다.

후속 issue 없음

수정 issue나 증거 issue를 만들지 않습니다.

끝

품질 통과나 실패가 아니라 "판단하지 않음"으로 끝납니다.

아직 판단 불가

URL, 브라우저 readback, work product, API 상태가 없어서 결론을 못 냅니다.

Needs evidence

필요한 evidence chip을 선택하고, 무엇을 확인할지 씁니다.

evidence_collecting

선택한 surface가 missing/blocking evidence로 기록됩니다.

Evidence issue 생성

Quality evidence required issue에서 증거를 수집합니다.

증거 해결 후 Queue 복귀

모든 blocking evidence가 해결되면 awaiting_review로 돌아와 다시 판정합니다.

기준을 만족함

목표, planned output, rubric, 증거가 맞고 더 보낼 수정이 없습니다.

Pass

왜 통과인지 짧게 남깁니다.

resolved_pass

이 review item의 사람 판정은 끝납니다.

후속 issue 없음

증거 수집이나 수정 흐름을 열지 않습니다.

끝

반복 가능한 통과 기준을 꼭 남길 때만 training case를 저장합니다.

고칠 점이 구체적임

산출물은 있고 판단도 가능하지만, 빠진 정의, 목표 불일치, 문구 결함 같은 rework가 필요합니다.

Request changes

무엇을 어디서 고칠지 rework 지시를 씁니다.

changes_requested

사람의 판정은 완료되고 수정 흐름이 열립니다.

Correction issue 생성

Quality correction required issue에서 담당자가 산출물을 고칩니다.

수정 후 재검토

고친 산출물이 돌아오면 다시 Pass 또는 추가 Request changes로 판단합니다.

차단성 품질 실패

산출물이 없거나, 목표와 완전히 다르거나, 단순 수정 요청보다 강하게 막아야 합니다.

Fail

왜 차단성 실패인지 씁니다.

anchor_candidate

실패 판정이 저장되고 학습 후보 상태가 됩니다.

Correction issue 생성

실패도 후속 처리 issue를 엽니다. 반복 규칙이면 training case 저장을 검토합니다.

재작성 또는 차단 처리

새 산출물이 생기면 별도 QA나 Quality review로 다시 확인합니다.

Record verdict 이후에 보이는 Evaluator learning case는 다른 흐름입니다.

위 표의 "저장 직후 상태"가 생기면 사람의 verdict는 이미 기록된 것입니다. Save training case는 그 verdict를 나중 자동 평가기가 배울 사례로 남길 때만 누릅니다.

목차 0. 전체 탭 루프 1. Queue 선택지 순서도 2. 기본 개념 3. Queue 카드 읽는 순서 4. 첫 판단 트리 5. 빠른 판단표 6. 버튼별 상세 흐름 7. Record verdict 이후 8. Evidence surfaces 9. Evaluator learning case 10. Anchors/Evaluators/Reports 11. 실제 예시 12. 흔한 오해

2. 기본 개념

Quality Board는 미션을 다시 실행하는 화면이 아니라, 산출물에 대한 사람의 품질 판정을 기록하는 화면입니다.

필수 흐름

판정 기록

Queue 항목을 보고 하나의 결론을 남깁니다. 결론은 Pass, Request changes, Needs evidence, Fail, Dismiss 중 하나입니다.

이 단계가 Quality reviewer가 반드시 해야 하는 작업입니다.

별도 흐름

Evaluator 학습 저장

이미 기록한 판정을 나중의 자동 평가기가 참고할 수 있는 사례로 저장합니다. 품질 판정 자체를 바꾸는 작업이 아닙니다.

반복될 규칙일 때만 저장합니다. 한 번뿐인 특수 상황이나 확신이 낮은 항목은 저장하지 않는 편이 낫습니다.

1 Queue 항목 발생 Final QA, Plan QA, Delivery QA, 수동 등록 등이 review item을 만듭니다.
2 사람이 대상 확인 Decision focus, 추천 행동, 증거, Mission 링크를 순서대로 봅니다.
3 Verdict 기록 하나의 버튼을 선택하고 Verdict note에 이유나 지시를 남깁니다.
4 처리 방향 결정 Pass는 완료, Request changes는 rework, Needs evidence는 증거 수집으로 갑니다.
5 학습 사례 저장 반복 규칙이면 제목을 다듬어 Save training case를 누릅니다.

3. Queue 카드 읽는 순서

화면의 모든 문장을 같은 무게로 읽으면 헷갈립니다. 아래 순서대로 보면 판단이 빨라집니다.

카드 제목

어떤 QA 단계에서 어떤 미션 산출물이 걸렸는지 보는 곳입니다. mission fb9d... 같은 긴 ID만으로는 사람이 알 수 없으므로, 화면에는 미션 이름과 회사 경로가 포함된 링크가 있어야 합니다.

Decision focus

가장 먼저 읽습니다. Target step, Planned output, Mission goal, Mismatch, Recommended action이 있으면 이 블록이 실제 판단의 중심입니다.

Recommended action

화면이 제안하는 기본 버튼입니다. 이유 문장과 맞아야 합니다. 이유가 이미 구체적인 수정 지시라면 Request changes가 기본입니다.

Raw QA excerpt

원래 QA가 남긴 긴 설명입니다. Decision focus가 충분하지 않을 때 세부 근거를 확인하는 곳입니다. 긴 원문 자체를 처음부터 해석하려고 하지 않아도 됩니다.

Evidence

실제 산출물, URL, 브라우저 확인, API 확인 같은 증거 상태입니다. 단, 구조화된 evidence가 비어 있어도 QA 원문이 이미 판단 근거를 주면 Needs evidence가 아닐 수 있습니다.

Mission context

미션 원문을 열어 맥락을 확인하는 링크입니다. 링크는 /<company>/missions/<mission-id> 형태여야 합니다. 미션 상태가 완료여도, 이 Quality item이 자동으로 Pass가 되는 것은 아닙니다.

4. 첫 판단 트리

버튼을 고르기 전 아래 질문을 위에서 아래로 답합니다.

이 항목이 중복, 오래된 항목, smoke/test, 현재 판단 대상이 아닌가?

예를 들어 이미 같은 건이 처리됐거나, 테스트용 항목이라 실제 품질 판단 대상이 아니면 여기서 끝냅니다.

Dismiss

Verdict note에 왜 제외하는지만 남깁니다.

현재 화면의 이유와 증거만으로 판단할 수 없는가?

URL이 열리는지, work product가 존재하는지, 브라우저 readback이 필요한데 없어서 결론을 못 내리는 경우입니다.

Needs evidence

필요한 evidence surface chip을 선택하고, 무엇을 확인해야 하는지 적습니다.

대상 산출물이 기준을 만족하는가?

목표, 계획, rubric, 증거가 맞고 더 보낼 수정이 없으면 통과입니다.

Pass

이 Quality item은 완료됩니다.

산출물은 있지만 구체적으로 고칠 점이 있는가?

QA가 이미 missing glossary, plan mismatch, formatting defect처럼 수정할 내용을 특정했다면 여기에 해당합니다.

Request changes

rework 지시를 남깁니다. 새 증거 요청이 기본이 아닙니다.

단순 rework가 아니라 품질 실패로 막아야 하는가?

산출물이 없거나, 목표와 완전히 다르거나, 운영상 그대로 진행하면 안 되는 심각한 실패일 때입니다.

Fail

차단성 실패로 기록합니다.

5. 빠른 판단표

Queue에서 시간이 없을 때 이 표만 보고도 기본 선택을 할 수 있어야 합니다.

화면에서 보이는 상황 누를 버튼 Verdict note에 쓸 내용 Record verdict 뒤 상태
산출물이 기준을 만족하고, 더 고칠 점이 없음 Pass 왜 통과인지 한 문장으로 충분합니다. 사람의 품질 판단은 끝납니다.
산출물은 있고 판단도 가능하지만, 구체적으로 고쳐야 할 점이 있음 Request changes 무엇을 어디서 고칠지, rework로 보낼 이유를 씁니다. 판정은 끝났고, 수정 작업 흐름으로 넘어갑니다.
증거가 부족해 아직 Pass/Fail/Request changes를 판단할 수 없음 Needs evidence 어떤 증거 surface가 필요한지와 확인할 내용을 씁니다. 증거 수집 상태가 되고, 증거가 돌아오면 다시 판단합니다.
단순 수정 요청보다 강한 차단성 실패 Fail 왜 품질 실패로 막아야 하는지 씁니다. 실패 판정으로 기록됩니다.
중복, 오래됨, 테스트 항목, 현재 판단 대상 아님 Dismiss 왜 제외하는지 씁니다. 활성 Queue에서 제외됩니다.

6. 버튼별 상세 흐름

각 버튼은 같은 "기록" 버튼처럼 보여도 이후 흐름이 다릅니다.

Pass 흐름도

산출물이 기준을 만족한다고 판단할 수 있을 때의 종료 흐름입니다.

최종 상태: resolved_pass
Reviewer
1 Decision focus 확인

Target step, planned output, mission goal, mismatch를 읽습니다.

2 통과 조건 확인

목표와 rubric을 만족하고, 더 고칠 항목이 없다고 판단합니다.

3 Pass 클릭

Record Pass dialog가 열립니다.

4 Verdict note 입력

왜 통과인지 짧게 씁니다. 길게 쓸 필요는 없습니다.

5 Record verdict 클릭

이 항목에 대한 사람의 품질 판단은 끝납니다.

Quality Board
A verdict row 저장

판정값 pass와 note를 저장합니다.

B 상태 변경

review item 상태가 resolved_pass가 됩니다.

C 후속 issue 없음

correction issue나 evidence issue를 새로 만들지 않습니다.

D 화면 문구

Recorded verdict: Pass. This quality item is done.

다음 담당자
끝 추가 작업 없음

미션 전체나 다른 산출물은 별도 이슈가 있을 때만 봅니다.

선택 Evaluator learning은 보통 생략

반복 가능한 "통과 기준"을 꼭 남겨야 할 때만 저장합니다.

Request changes 흐름도

판단은 가능하고 산출물도 있지만, 구체적인 수정이 필요할 때의 rework 흐름입니다.

최종 상태: changes_requested
Reviewer
1 문제 근거 확인

Mismatch 또는 Raw QA excerpt에서 고칠 지점이 이미 특정되어 있는지 봅니다.

2 수정 요청 조건 확인

산출물은 존재하고, 증거 부족이 아니라 내용 결함이라고 판단합니다.

3 Request changes 클릭

Record Request changes dialog가 열립니다.

4 Verdict note 작성

무엇을, 어디서, 왜 고칠지 씁니다. 새 증거 요청을 기본으로 쓰지 않습니다.

5 Record verdict 클릭

사람의 판정은 여기서 완료됩니다.

Quality Board
A verdict row 저장

판정값 request_changes와 rework note를 저장합니다.

B 상태 변경

review item 상태가 changes_requested가 됩니다.

C correction issue 생성

Quality correction required: ... 형태의 수정 issue가 todo로 열립니다.

D 화면 문구

The correction flow is open; no more Quality input is required here unless this verdict was wrong.

Producer / Owner
6 수정 issue 처리

담당 step이나 산출물 담당자가 note에 적힌 결함을 수정합니다.

7 수정 산출물 제출

수정된 work product나 보고서를 다시 QA로 보냅니다.

8 다시 판단

수정 후에는 Pass, 다시 Request changes, Fail 중 하나로 재판정합니다.

Evaluator 학습
선택 Case title 확인

반복 규칙이면 제목을 "무엇을 배워야 하는가"로 다듬습니다.

선택 Save training case

저장하면 future evaluator가 비슷한 결함을 더 잘 잡는 재료가 됩니다.

Needs evidence 흐름도

증거가 부족해 아직 품질 결론을 낼 수 없을 때의 대기 및 복귀 흐름입니다.

최종 상태: evidence_collecting → awaiting_review
Reviewer
1 판단 불가 이유 확인

내용 결함이 아니라 proof surface가 없어서 결론을 못 내리는지 확인합니다.

2 Needs evidence 클릭

Record Needs evidence dialog가 열립니다.

3 Verdict note 작성

무슨 증거가 없어서 판단할 수 없는지 씁니다.

4 Evidence chip 선택

public_url, browser_readback, work_product, api_probe 중 필요한 항목을 고릅니다.

5 Record verdict 클릭

chip이 하나도 없으면 저장할 수 없습니다.

Quality Board
A verdict row 저장

판정값 needs_evidence와 evidence request note를 저장합니다.

B 상태 변경

review item 상태가 evidence_collecting이 됩니다.

C evidence refs 생성

선택한 surface가 missing과 blocking으로 기록됩니다.

D evidence issue 생성

Quality evidence required: ... issue가 todo로 열립니다.

Evidence 담당자
6 증거 수집

URL, 브라우저, work product, API 상태를 실제로 확인합니다.

7 Evidence status 기록

verified, failed, insufficient, missing, stale, not_applicable 중 하나로 기록합니다.

8 자동 분기

blocking unresolved가 남았는지에 따라 상태가 갈립니다.

분기 A: 아직 unresolved evidence가 남음

하나라도 missing, failed, insufficient, stale 또는 blocking 상태면 계속 evidence_collecting입니다. 사람은 아직 최종 verdict를 내리지 않습니다.

분기 B: 모든 blocking evidence가 해결됨

상태가 awaiting_review로 돌아옵니다. 이때 사람이 다시 Queue에서 보고 Pass, Request changes, Fail, Dismiss 중 하나로 최종 판단합니다.

Fail 흐름도

수정 요청보다 강하게 품질 실패로 기록해야 할 때의 차단 및 학습 후보 흐름입니다.

최종 상태: anchor_candidate
Reviewer
1 차단성 실패 확인

산출물이 없거나, 목표와 완전히 다르거나, 그대로 진행하면 안 되는지 봅니다.

2 Fail 클릭

Record Fail dialog가 열립니다.

3 Verdict note 작성

왜 차단성 실패인지, 어느 산출물 또는 step이 실패인지 씁니다.

4 Record verdict 클릭

사람의 실패 판정은 저장됩니다.

Quality Board
A verdict row 저장

판정값 fail과 실패 이유를 저장합니다.

B 상태 변경

review item 상태가 anchor_candidate가 됩니다.

C correction issue 생성

fail도 수정 또는 후속 처리 issue를 엽니다.

D 화면 문구

Recorded verdict: Fail. No more verdict input is required.

Owner / Rework
5 실패 원인 처리

산출물 재작성, 목표 재정렬, 누락 산출물 생성 등 후속 조치를 합니다.

6 재검토

새 산출물이 생기면 별도 Quality review 또는 QA로 다시 확인합니다.

Evaluator 학습
선택 Save training case

이 실패가 반복 규칙이면 anchor로 승격합니다.

선택 candidate evaluator 생성

저장하면 evaluator candidate version과 replay run이 만들어집니다.

선택 Evaluators 탭에서 replay

replay가 passed가 되어야 production evaluator로 승격할 수 있습니다.

Dismiss 흐름도

이 항목이 현재 품질 판단 대상이 아닐 때 활성 Queue에서 제외하는 흐름입니다.

최종 상태: dismissed
Reviewer
1 제외 조건 확인

중복, 오래됨, smoke/test, 대상 아님 중 하나인지 확인합니다.

2 Dismiss 클릭

Record Dismiss dialog가 열립니다.

3 Verdict note 작성

왜 판단하지 않고 제외하는지 씁니다.

4 Record verdict 클릭

Queue 처리에서 제외됩니다.

Quality Board
A verdict row 저장

판정값 dismissed와 제외 이유를 저장합니다.

B 상태 변경

review item 상태가 dismissed가 됩니다.

C 후속 issue 없음

correction issue나 evidence issue를 만들지 않습니다.

다음 담당자
끝 품질 통과/실패가 아님

Dismiss는 "이 항목을 판단하지 않는다"는 뜻입니다. 산출물이 좋다는 뜻이 아닙니다.

선택 Evaluator learning은 보통 생략

중복이나 오래된 항목은 evaluator 학습 사례로 저장하지 않는 편이 좋습니다.

Pass

통과
누르는 경우
  • 대상 산출물이 목표와 rubric을 만족합니다.
  • 필요한 증거가 충분합니다.
  • 더 보낼 수정 지시가 없습니다.
입력할 내용

짧게 써도 됩니다. 예: Pass RES-614. The report defines required terms and maps claims to sources.

누른 뒤 흐름

Quality reviewer 입장에서는 끝난 상태입니다. 학습 사례 저장은 보통 필요하지 않습니다.

주의

Mission이 completed 또는 pass처럼 보여도, 그것만으로 Quality item을 Pass하면 안 됩니다. Queue가 묻는 것은 미션 전체가 아니라 이 카드의 target output입니다.

Request changes

수정 요청
누르는 경우
  • 산출물은 존재합니다.
  • 화면의 이유만으로 문제를 판단할 수 있습니다.
  • 빠진 항목, 목표 불일치, operator-facing 문구 문제처럼 고칠 지점이 구체적입니다.
입력할 내용

rework 지시를 씁니다. "증거가 필요하다"가 아니라 "무엇을 고쳐라"가 중심입니다.

누른 뒤 흐름

사람의 판정은 기록됩니다. 이후에는 생산자나 담당 step이 산출물을 수정해야 합니다.

Request changes for RES-614 report draft.
Reason: The report uses PoC, AppSec, and ToS in operator-facing sections, but the glossary does not define them.
Action: Add definitions for PoC, AppSec, and ToS, then return the report for QA. No fresh evidence is needed because the work product already exists.
핵심 구분

Request changes는 "판단을 못 하겠다"가 아닙니다. 이미 판단했고, 결론이 "수정해서 다시 가져와라"인 경우입니다.

Needs evidence

증거 필요
누르는 경우
  • 현재 화면만으로는 판단을 내릴 수 없습니다.
  • 특정 proof surface가 빠져 있습니다.
  • 증거가 돌아오면 다시 Pass, Request changes, Fail 중 하나를 판단해야 합니다.
입력할 내용

Verdict note에는 "무슨 증거가 없어서 판단할 수 없는지"를 씁니다. 그리고 필요한 chip을 선택합니다.

누른 뒤 흐름

항목은 evidence collecting 상태로 갑니다. 증거가 채워지면 사람이 다시 판단해야 합니다.

Need more evidence before judging RES-614.
Missing evidence: browser_readback and work_product.
Action: confirm the registered report file resolves and the published report renders in the browser, then return this item to human review.
주의

긴 QA 원문이 어렵다는 이유만으로 Needs evidence를 누르지 않습니다. 원문 안에 이미 구체적인 실패 이유가 있으면 보통 Request changes입니다.

Fail

실패
누르는 경우
  • 산출물이 없거나 사용할 수 없습니다.
  • 목표와 완전히 다릅니다.
  • 단순 rework보다 강하게 품질 실패로 막아야 합니다.
입력할 내용

실패 이유와 차단해야 하는 이유를 씁니다. 재사용할 규칙이면 학습 사례로 저장할 수 있습니다.

누른 뒤 흐름

실패 판정으로 남습니다. 이후 처리 정책은 이 실패가 어떤 종류인지에 따라 달라집니다.

Request changes와 차이

수정해서 다시 가져오면 될 정도면 Request changes가 더 맞습니다. Fail은 산출물의 신뢰성이 크게 깨졌거나 그대로 흐름을 진행하면 안 되는 경우에 씁니다.

Dismiss

제외
누르는 경우
  • 중복 항목입니다.
  • 이미 오래된 항목입니다.
  • 실제 품질 판단 대상이 아닙니다.
입력할 내용

왜 제외하는지 짧게 씁니다. 예: Duplicate of the already reviewed RES-614 quality item.

누른 뒤 흐름

활성 Queue에서 빠집니다. 산출물 통과나 실패를 의미하지 않습니다.

7. Record verdict 이후 무엇이 끝난 것인가

사용자가 가장 헷갈리는 지점은 "버튼을 눌렀는데 또 뭔가 나타나는" 순간입니다.

끝난 것

사람의 Quality 판정

Recorded verdict: Request changes처럼 보이면, 이 카드에 대한 사람의 판정은 이미 저장된 것입니다. 같은 버튼을 다시 누르는 것은 보통 필요하지 않습니다.

Edit verdict는 방금 판정이 잘못됐을 때만 씁니다.

남은 것

판정 결과에 따른 후속 작업

Request changes라면 산출물 담당자가 수정해야 합니다. Needs evidence라면 증거를 모아야 합니다. Pass나 Dismiss는 보통 Queue 작업이 끝납니다.

왜 Evaluator learning case가 계속 보이나요?

그 패널은 "방금 판정을 나중 평가기가 배울 사례로 남길 것인가"를 묻는 별도 단계입니다. 품질 판정 완료와 별개입니다. 저장하지 않아도 방금 verdict가 취소되지는 않습니다.

8. Evidence surfaces

Needs evidence를 선택할 때는 자유 입력 대신 chip으로 필요한 증거 표면을 고릅니다.

Chip 뜻 선택하는 경우 돌아온 뒤 판단
public_url 게시된 URL이 존재하고 접근 가능한지 산출물이 URL 형태인데 실제 공개 페이지 확인이 없을 때 URL이 열리고 대상 내용이 맞으면 다음 verdict를 판단합니다.
browser_readback 브라우저에서 렌더링된 화면을 읽어 확인했는지 HTML, 보고서, UI처럼 렌더링 결과가 품질 판단에 중요할 때 브라우저에서 보이는 문구, 링크, 레이아웃을 기준으로 판단합니다.
work_product 등록된 산출물 파일이나 work product가 실제로 해석되는지 시스템은 산출물이 있다고 말하지만 파일, 경로, 등록 상태가 불확실할 때 산출물이 존재하면 내용 품질을 판단하고, 없으면 실패 또는 수정 요청을 판단합니다.
api_probe API나 데이터베이스 상태가 실제로 맞는지 화면 말고 런타임 상태, 등록 여부, 처리 상태가 핵심일 때 상태가 확인되면 그 상태를 근거로 verdict를 기록합니다.

Evidence status를 기록할 때

기존 evidence row에는 verified, failed, insufficient, missing, stale, not_applicable 상태를 기록할 수 있습니다.

판단 기준

missing, failed, insufficient, stale가 blocking이면 아직 해결되지 않은 증거입니다. verified는 판단 가능한 증거입니다.

9. Evaluator learning case

이 단계는 "이번 verdict를 자동 평가기가 나중에 배울 규칙으로 저장할지"를 결정합니다.

저장하는 경우

반복 가능한 규칙일 때

  • 앞으로도 비슷한 산출물이 같은 기준으로 걸릴 가능성이 높습니다.
  • 한 문장 제목으로 재사용 규칙을 설명할 수 있습니다.
  • 자동 평가기가 다음에 같은 실수를 더 잘 잡아야 합니다.
저장하지 않는 경우

특수하거나 불확실할 때

  • 이번 한 번만 해당하는 맥락입니다.
  • 판정 근거가 불확실하거나 noisy합니다.
  • 제목이 단순히 항목 이름만 반복하고 규칙을 담지 못합니다.
필드/버튼 무엇을 하는가 좋은 입력 피해야 할 입력
Case title 나중 evaluator가 배울 사례 제목입니다. 자동으로 채워지지만 사람이 수정할 수 있습니다. Request changes: operator-facing jargon must be defined in the glossary. tech-scout thing, fb9d5e0c, bad report
Save training case 기록된 verdict를 anchor case로 저장합니다. 이후 evaluator candidate와 replay 흐름의 재료가 됩니다. 반복 규칙을 남길 때 누릅니다. 품질 verdict를 끝내려고 누르는 버튼으로 오해하면 안 됩니다.

10. Anchors, Evaluators, Reports 탭 흐름

Queue에서 verdict를 기록한 뒤에도 Quality Board에는 세 가지 운영 흐름이 남습니다. Anchors는 사람이 남긴 판단 사례, Evaluators는 자동 평가기 후보, Reports는 현재 상태 기록입니다.

Anchors 흐름도

Anchor는 "이번 사람 판정을 다음에도 같은 기준으로 적용하라"는 재사용 사례입니다.

목표: reusable precedent 확인
Queue
1 Verdict 기록 완료

Pass, Request changes, Needs evidence, Fail, Dismiss 중 하나가 먼저 저장되어 있어야 합니다.

2 반복 규칙인지 판단

비슷한 산출물이 다음에도 같은 기준으로 걸려야 하면 학습 사례로 저장합니다.

3 Case title 다듬기

항목 이름이 아니라 evaluator가 배울 규칙을 제목에 씁니다.

4 Save training case

이 버튼을 눌러 anchor를 만듭니다. Queue verdict 자체를 바꾸지는 않습니다.

System
A Anchor case 생성

human verdict, failure type, evidence refs, title이 anchor로 저장됩니다.

B Candidate evaluator 생성

anchor 저장과 함께 evaluator candidate version이 만들어집니다.

C Queued replay run 생성

candidate가 과거 사례를 통과하는지 확인할 replay run이 대기 상태로 생깁니다.

D Improvement issue 생성

자동 평가기 개선 작업 issue가 열립니다.

Anchors 탭
5 Anchor 목록 확인

title, failure type, human verdict, evidence refs를 봅니다.

6 Precedent 문장 확인

비슷한 future output을 같은 방식으로 판단할 기준인지 봅니다.

7 Reflected in evaluator 확인

yes면 evaluator version에 연결됐습니다. not yet이면 candidate 생성 또는 동기화가 필요합니다.

Anchors 탭에서 사용자가 직접 누르는 주요 버튼은 없습니다.

Anchor 생성은 Queue의 Save training case에서 일어납니다. Anchors 탭은 저장된 precedent가 정확한지, evaluator에 반영됐는지 읽는 화면입니다.

Evaluators 흐름도

Evaluator는 anchor를 보고 만들어진 자동 평가기 후보입니다. 바로 운영에 올리지 않고 replay로 검증합니다.

목표: candidate → production
시작 조건
1 Anchor가 있어야 함

Queue에서 Save training case를 저장해야 candidate evaluator가 생깁니다.

2 Candidate versions 확인

source anchor, version status, 연결된 replay runs를 봅니다.

3 Replay 필요 여부 확인

버튼이 replay required면 아직 production 승격 조건을 만족하지 못했습니다.

Evaluator 탭
4 Replay 클릭

candidate evaluator를 과거 anchor case에 적용합니다.

5 Replay result 저장

regression이 없으면 passed, 있으면 failed입니다.

6 passed 확인

passed replay가 하나 이상 있어야 production 승격 버튼이 활성 의미를 갖습니다.

7 Promote to production

운영 자동 평가기로 승격합니다.

System
A Evaluator status 변경

candidate version 상태가 production이 됩니다.

B Source review item 갱신

출처 review item은 evaluator_promoted로 표시됩니다.

C Reports에 반영 가능

이후 report를 다시 생성하면 evaluator/anchor 상태가 최신 summary에 반영됩니다.

Replay failed

candidate evaluator가 과거 사례를 안정적으로 재현하지 못한 것입니다. production 승격하지 말고 evaluator 개선 issue를 봅니다.

Replay passed

candidate가 최소 조건을 통과했습니다. 운영에 반영할 준비가 된 경우에만 production으로 승격합니다.

Reports 흐름도

Reports는 현재 Quality Board 상태를 날짜별로 저장하는 요약입니다. 판정이나 evidence가 바뀌면 다시 생성해야 합니다.

목표: current dated summary
사용자
1 Queue 상태 확인

open review item, evidence collecting, changes requested가 남아 있는지 봅니다.

2 Anchors/Evaluators 상태 확인

anchor coverage gap과 evaluator candidate 상태가 report에 들어갑니다.

3 Generate today's report

오늘 날짜 report를 생성하거나 이미 있으면 갱신합니다.

System
A Summary 계산

pendingReviewItems, needsEvidenceOutstanding, anchorCoverageGaps, evidence stats를 계산합니다.

B Report status 결정

미해결 evidence가 있으면 needs_attention, 아니면 generated입니다.

C 같은 날짜 report upsert

오늘 report가 있으면 덮어 갱신하고, 없으면 새로 만듭니다.

Reports는 판단을 대신하지 않습니다.

Report는 현재 상태를 저장한 요약입니다. Queue 항목을 Pass/Request changes로 바꾸거나 evidence를 해결하는 실제 처리는 Queue와 Evidence 흐름에서 해야 합니다.

11. 실제 예시: Final QA / purpose-fitness failure - 2026-06-29 tech-scout

보고서 용어집 누락 사례를 화면에서 봤을 때의 판단 흐름입니다.

권장 행동

이 케이스의 기본 행동은 Request changes입니다. 이유는 work product가 존재하고, QA가 이미 구체적인 수정 이유를 제시했기 때문입니다: operator-facing 섹션에 PoC, AppSec, ToS가 나오지만 glossary 정의가 빠졌습니다.

왜 Needs evidence가 아닌가

"No structured evidence"라는 문장이 보일 수 있지만, 긴 QA 원문 안에는 이미 판단 가능한 근거가 있습니다. 산출물이 존재하고 파일도 해석되며, 실패 이유는 "세 용어 정의 누락"으로 특정되어 있습니다.

따라서 새 증거를 요청하기보다 수정 지시를 보내는 것이 자연스럽습니다.

왜 Mission pass만 보면 안 되나

Mission 상태는 전체 작업 흐름의 상태입니다. Queue의 이 항목은 RES-614 보고서 초안이 operator-facing 기준을 만족하는지 묻습니다. 미션이 완료되어도 산출물의 세부 품질 결함은 별도로 rework될 수 있습니다.

Verdict note 예시

Request changes for RES-614 report draft.
Reason: The report exists and mostly passes, but glossary definitions for PoC, AppSec, and ToS are missing while those terms appear in operator-facing sections.
Action: Add definitions for PoC, AppSec, and ToS to the glossary, then return the report for QA. Do not request fresh evidence unless the corrected report cannot be resolved.

Evaluator learning case 제목 예시

Request changes: operator-facing jargon must be defined in the glossary.

12. 흔한 오해

Quality Board를 처음 볼 때 반복해서 헷갈리는 지점입니다.

오해 1. "Evidence가 비어 있으면 무조건 Needs evidence다"

아닙니다. QA 원문이 이미 구체적인 결함과 수정 방향을 제시하면 Request changes가 맞을 수 있습니다.

오해 2. "Request changes를 누른 뒤 또 뭔가 보이면 아직 안 끝난 것이다"

사람의 verdict는 끝났습니다. 뒤에 보이는 Evaluator learning case는 별도 학습 저장 단계입니다.

오해 3. "Mission이 pass면 Quality item도 pass다"

아닙니다. 미션 상태와 개별 산출물 품질 판단은 다릅니다. Queue 카드의 target output을 기준으로 판단합니다.

오해 4. "Fail과 Request changes는 비슷하다"

다릅니다. 고쳐서 다시 가져오라는 뜻이면 Request changes입니다. 차단성 품질 실패로 기록해야 하면 Fail입니다.

오해 5. "Case title에는 아무거나 써도 된다"

아닙니다. evaluator가 배울 재사용 규칙을 담아야 합니다. 항목 이름보다 "어떤 기준을 기억해야 하는지"가 중요합니다.

요약: Queue에서는 먼저 사람이 판단 가능한지 확인합니다. 판단 가능하고 고칠 점이 구체적이면 Request changes, 판단 자체가 불가능하면 Needs evidence, 기준을 만족하면 Pass, 차단성 실패면 Fail, 판단 대상이 아니면 Dismiss입니다.
