#### 1. Plugin SDK 의존의 위험성 — upstream이 API를 바꾸면?

**의견:** 설계 원칙에서 "upstream 업데이트 영향 없음" 및 "서버 소스 포크 없음"을 명시하고, 모든 커스텀 로직을 Plugin으로 구현하려는 의도는 매우 긍정적입니다. `ctx.events.on`, `ctx.issues.create/update`, `ctx.agents.invoke`, `ctx.state.set/get` 등 언급된 SDK 기능들은 Paperclip Plugin 시스템의 핵심적인 부분으로 보입니다.

**우려사항:**
*   **SDK API 안정성:** Plugin SDK 자체의 API가 변경될 경우, 플러그인은 upstream에 직접적으로 종속되지 않더라도 SDK 변경에 의해 영향을 받을 수 있습니다. 현재 Paperclip Plugin SDK의 API 변경 정책이나 안정성에 대한 정보가 있다면 더 좋습니다. 메이저 버전 업데이트 시 파괴적 변경(breaking changes)이 발생할 가능성은 항상 존재하며, 이는 플러그인 유지보수 부담으로 이어질 수 있습니다.
*   **버전 관리:** Plugin이 특정 SDK 버전에 의존하게 될 경우, Paperclip 서버와 Plugin 간의 호환성 문제가 발생할 수 있습니다. Plugin의 manifest에 최소/최대 SDK 버전을 명시하는 등의 전략이 필요할 수 있습니다.

#### 2. Workflow Engine의 상태 관리 — ctx.state만으로 DAG 실행 상태를 안전하게 관리할 수 있는가?

**의견:** `ctx.state`를 워크플로우 정의 및 실행 상태의 중앙 저장소로 사용하는 것은 간결하며 Plugin 기반 접근 방식에 적합합니다. `issue.updated` 이벤트에 기반한 트리거 방식도 효율적입니다.

**우려사항:**
*   **동시성 및 일관성:** 여러 `issue.updated` 이벤트가 거의 동시에 발생하여 `ctx.state`에 대한 동시 쓰기(concurrent writes)가 발생할 경우, 상태 일관성 문제가 발생할 수 있습니다. `ctx.state.set/get`이 내부적으로 어떤 동시성 제어 메커니즘(예: 락, 낙관적 잠금)을 제공하는지 확인해야 합니다. 일관성이 깨지면 워크플로우가 잘못된 상태로 진입하거나, 특정 스텝이 두 번 실행되거나 누락될 수 있습니다.
*   **트랜잭션 보장:** 워크플로우의 상태 변화(예: 다음 스텝 `todo`로 변경)는 여러 연산(상태 읽기, 검증, 상태 쓰기)을 포함할 수 있습니다. 이 과정에서 플러그인이 비정상 종료되거나 오류가 발생하면 `ctx.state`가 일관되지 않은 중간 상태로 남을 수 있습니다. `ctx.state`가 트랜잭션적 속성을 얼마나 보장하는지 확인하거나, 플러그인 내에서 복구 가능한 방법을 설계해야 합니다.
*   **스케일링:** 대량의 워크플로우와 스텝이 동시에 실행될 때 `ctx.state`의 성능 병목 현상이 발생할 가능성이 있습니다. `ctx.state`의 내부 구현(저장 방식, 쿼리 효율성 등)에 대한 이해가 필요합니다.
*   **DAG 무결성:** `dag-engine.ts`가 `ctx.state`에 저장된 워크플로우 정의(특히 `dependsOn`)를 해석할 때, 순환 의존성(circular dependencies)이나 잘못된 정의에 대한 검증 로직이 필요합니다.

**제안:**
*   `ctx.state`가 특정 버전의 Paperclip 서버에 종속될 수 있으므로, 장기적으로는 워크플로우 상태 관리를 위한 전용 데이터 저장소(예: RDBMS)를 사용하는 것을 고려해 볼 수 있습니다. 이는 복잡성을 증가시키지만, 더 높은 신뢰성과 유연성을 제공합니다.
*   `ctx.state`에 대한 모든 쓰기 작업을 큐잉하고 단일 처리 스레드에서 순차적으로 처리하는 방식(예: 액터 모델)을 플러그인 내부에 구현하여 동시성 문제를 완화할 수 있습니다.
*   워크플로우 스텝의 상태가 변경될 때마다 이전 상태와 새로운 상태를 모두 기록하여 디버깅 및 감사 추적을 용이하게 합니다.

#### 3. heartbeat OFF 전략의 빈틈 — 이벤트 유실 시 에이전트가 영원히 잠들 수 있는가?

**의견:** heartbeat를 제거하고 이벤트 기반 wakeup에만 의존하는 것은 리소스 낭비(특히 토큰)를 줄이는 좋은 전략입니다.

**우려사항:**
*   **이벤트 유실 위험:** `issue.created` 또는 `issue.updated` 이벤트가 Paperclip 내부의 이벤트 버스 문제, 네트워크 오류, 플러그인 재시작, 예외 처리 미흡 등으로 인해 유실될 경우, 해당 이슈에 할당된 에이전트가 절대로 깨어나지 않거나 다음 워크플로우 스텝이 트리거되지 않는 치명적인 문제가 발생할 수 있습니다. "영원히 잠들 수 있는가?"라는 질문에 "그렇다"고 답할 수 있는 잠재적 취약점입니다.

**제안:**
*   **조정자(Reconciler) 패턴:** 주기적으로 실행되는 "reconciler" 또는 "감독(supervisor)" 루틴을 도입합니다. 이 루틴은 `todo` 상태로 남아있지만 일정 시간 동안 `wakeup` 이벤트가 발생하지 않은 이슈를 스캔하여 강제로 `wakeup`을 재트리거하거나 관리자에게 알림을 보냅니다. 이는 `ctx.jobs` (cron)을 활용하여 구현할 수 있습니다.
*   **이벤트 내구성 강화:** Paperclip 내부 이벤트 버스가 이벤트 유실 방지를 위한 내구성(예: 메시지 큐, 영속화)을 얼마나 제공하는지 확인해야 합니다.
*   **재시도 로직:** `wakeup` 트리거 로직에 재시도 메커니즘을 추가하여 일시적인 오류로 인한 유실을 방지합니다.

#### 4. CLI Registry의 보안 — 에이전트가 허가 외 CLI를 실행하는 것을 실제로 막을 수 있는가?

**의견:** CLI Registry는 에이전트의 권한을 제어하여 시스템 무결성을 유지하려는 좋은 시도입니다. 그러나 현재 문서의 설명만으로는 "실제로 막을 수 있는가?"에 대한 확신이 부족합니다.

**우려사항:**
*   **강제성 부족:** "instructions에 자동 주입" 및 "위 도구 외에는 실행 권한이 없습니다."라는 문구는 에이전트에게 정보를 제공하는 것이지, 기술적으로 실행을 제어하는 메커니즘으로 보이지 않습니다. 만약 에이전트가 지시사항을 무시하고 내부적으로 다른 CLI 명령어를 구성하여 실행하려고 시도한다면, 현재 설명된 방식으로는 막을 수 없습니다.
*   **실행 환경 제어:** `command` 필드에 `'python scripts/data-collection/collect.py'`와 같은 스크립트 경로가 포함됩니다. 에이전트가 이 스크립트 파일을 직접 수정하거나, 스크립트가 실행될 때 외부 환경 변수를 조작하여 예상치 못한 동작을 유발할 수 있는가? `workingDirectory`와 `env` 필드도 잠재적인 보안 취약점이 될 수 있습니다. 에이전트의 CLI 실행 환경에 대한 강력한 샌드박싱이 없다면 위험합니다.

**제안:**
*   **실행 가로채기(Execution Interception):** `agent.run.command` 호출을 플러그인 레벨에서 가로채(intercept)야 합니다. 플러그인은 에이전트가 요청한 CLI 명령어가 `AgentToolGrant`에 따라 허용된 도구 목록에 있는지 확인하고, 인자 스키마 (`argsSchema`)에 따라 인자를 검증한 후에만 실제 CLI 실행을 Paperclip 서버에 요청해야 합니다. 이렇게 해야만 허가되지 않은 CLI 실행을 *실제로* 막을 수 있습니다.
*   **스크립트 무결성:** `command`로 지정된 스크립트 파일의 해시(hash)를 `Tool` 정의에 포함하여, 스크립트가 실행되기 전에 변경되지 않았음을 검증하는 방안을 고려할 수 있습니다.
*   **샌드박싱:** 에이전트가 실행하는 CLI 명령어에 대해 엄격한 샌드박싱 환경(예: 컨테이너화, 제한된 사용자 권한)을 제공하여, 파일 시스템 접근이나 네트워크 통신을 제한해야 합니다. 이는 Paperclip 서버 수준에서 제공되어야 할 기능입니다.

#### 5. 단일 장애점 — Paperclip 서버 다운 시 모든 것이 멈추는 문제

**의견:** 이는 Paperclip 중앙화 아키텍처의 필연적인 결과이며, 설계 문서가 주로 내부 운영 플랫폼 기능에 초점을 맞추고 있으므로 이 자체를 해결하는 것은 범위를 벗어납니다.

**우려사항:**
*   없음. (설계 범위 외)

**제안:**
*   Paperclip 서버의 고가용성(High Availability) 및 재해 복구(Disaster Recovery) 전략이 별도로 수립되어야 합니다. 이는 전체 시스템의 견고성을 위해 중요합니다.

#### 6. 실행 순서 경합 — 동시에 여러 workflow가 같은 에이전트를 호출하면?

**의견:** 워크플로우 엔진이 `issue.updated` 이벤트에 따라 다음 스텝의 이슈를 `todo` 상태로 변경하고 에이전트를 `wakeup` 시키는 방식은 잘 정의되어 있습니다. "병렬 실행"은 워크플로우 내 스텝의 병렬 처리에 관한 것입니다.

**우려사항:**
*   **에이전트 워크로드 관리:** 만약 여러 워크플로우 또는 단일 워크플로우 내 병렬 스텝들이 동시에 특정 에이전트 (`assigneeAgentId`)에게 이슈를 할당하고 `wakeup`을 트리거할 경우, 해당 에이전트의 처리 능력(concurrency)을 초과할 수 있습니다.
    *   에이전트가 한 번에 하나의 이슈만 처리할 수 있다면, 다른 이슈들은 처리되지 않고 대기 상태로 남을 수 있습니다.
    *   에이전트가 여러 이슈를 동시에 처리할 수 있다면, 에이전트 자체의 동시성 제어 로직이 견고해야 하며, 공유 자원에 대한 경합(race condition) 문제가 발생하지 않도록 설계되어야 합니다.
*   **백프레셔(Backpressure):** 에이전트의 처리 속도보다 이슈 발생 속도가 빠를 경우, `todo` 상태의 이슈 큐가 과도하게 쌓여 시스템 전반의 지연을 유발할 수 있습니다.

**제안:**
*   **에이전트별 처리량 설정:** 각 에이전트의 처리 가능한 동시 이슈 개수를 설정할 수 있도록 하고, 워크플로우 엔진은 이를 고려하여 에이전트 할당 및 `wakeup`을 제어해야 합니다. 예를 들어, 에이전트가 `max_concurrent_tasks` 개수만큼의 `in_progress` 이슈를 가지고 있다면, 더 이상 `todo` 이슈를 할당하지 않도록 합니다.
*   **이슈 큐잉 및 우선순위:** Paperclip의 이슈 할당/스케줄링 메커니즘이 에이전트의 워크로드를 얼마나 효율적으로 관리하는지 확인해야 합니다. 필요한 경우 워크플로우 스텝에 우선순위 속성을 추가하여 중요한 작업부터 처리되도록 할 수 있습니다.
*   **실패 전략:** 에이전트가 과부하로 인해 이슈를 처리하지 못할 경우 (`timeoutMinutes` 발생 등), 해당 스텝에 대한 `onFailure` 정책이 적절하게 작동하도록 해야 합니다.

#### 7. 비용 제어 — workflow가 무한 루프에 빠지면?

**의견:** 각 `WorkflowStep`에 `timeoutMinutes`가 정의된 것은 개별 스텝의 비용을 제어하는 좋은 방법입니다.

**우려사항:**
*   **논리적 무한 루프:** `dependsOn`으로 구성된 DAG는 기술적으로 순환 의존성을 방지하지만, 에이전트가 특정 스텝에서 의도치 않게 이전 스텝을 다시 트리거하는 이슈를 생성하거나, 특정 조건이 계속 충족되지 않아 스텝이 반복적으로 재시도되는 "논리적 무한 루프"에 빠질 수 있습니다. 이 경우 개별 스텝의 타임아웃은 짧더라도 전체 워크플로우는 무한정 실행될 수 있습니다.
*   **워크플로우 단위 비용 제어 부재:** `Workflow` 전체에 대한 총 실행 시간 제한(`workflowTimeoutMinutes`)이나 총 비용 제한(`maxCost`) 메커니즘이 없습니다. 특정 워크플로우가 예상보다 훨씬 많은 비용(토큰 사용, 컴퓨팅 시간)을 소모하는 것을 방지할 수 없습니다.

**제안:**
*   **워크플로우 타임아웃:** `Workflow` 레벨에 `timeoutMinutes` 속성을 추가하여, 전체 워크플로우가 이 시간을 초과하면 자동으로 중단되도록 합니다.
*   **비용 모니터링 및 알림:** 각 워크플로우 실행에 대한 토큰 사용량이나 컴퓨팅 비용을 추적하고, 특정 임계값을 초과할 경우 관리자에게 알림을 보내는 기능을 도입합니다. 이는 Paperclip의 비용 관리 기능과 연동될 수 있습니다.
*   **최대 재시도 횟수:** `onFailure: "retry"` 정책을 사용하는 경우, 무한 재시도를 방지하기 위해 `maxRetries` 속성을 추가해야 합니다.

#### 8. 테스트 전략 — workflow DAG를 어떻게 테스트할 것인가?

**의견:** 설계 문서에는 테스트 전략이 명시적으로 포함되어 있지 않습니다. Workflow Engine은 시스템의 핵심적인 부분이므로 견고한 테스트 전략이 필수적입니다.

**우려사항:**
*   없음. (누락된 우려사항)

**제안:**
*   **단위 테스트 (Unit Tests):** `dag-engine.ts`의 `dependsOn` 해석, 병렬 실행 조건, `join` 로직, `onFailure` 정책 등에 대한 단위 테스트를 철저히 수행합니다. 특히 경계 조건(edge cases), 빈 의존성, 복잡한 `join` 등을 다룹니다.
*   **통합 테스트 (Integration Tests):** `issue.updated` 이벤트가 트리거될 때 플러그인이 `ctx.state`를 업데이트하고 `ctx.issues.create/update`, `ctx.agents.invoke`를 호출하는 전체 흐름을 모의(mock)하여 테스트합니다. Paperclip SDK 호출을 모의하여 실제 서버 없이도 테스트 가능하도록 합니다.
*   **종단 간 테스트 (End-to-End Tests):** 실제 Paperclip 서버 환경에서 워크플로우 정의를 배포하고, 더미 에이전트나 간단한 스크립트를 사용하여 이슈 상태 변화를 시뮬레이션하면서 워크플로우가 의도대로 동작하는지 확인합니다. "가즈아 일일 루틴"과 같은 실제 예시 워크플로우를 테스트 시나리오로 활용합니다.
*   **실패 시나리오 테스트:** 네트워크 오류, 에이전트 실패, 타임아웃 등 다양한 실패 시나리오에서 `onFailure` 정책이 올바르게 작동하는지 테스트합니다.
*   **성능 테스트:** 대규모 워크플로우(많은 스텝, 많은 병렬 실행)에 대한 성능 테스트를 수행하여 `ctx.state`의 병목 현상 여부를 확인합니다.

#### 9. 그 외 누락된 우려사항

*   **관측 가능성(Observability) 및 모니터링:**
    *   워크플로우의 전체 진행 상황을 시각적으로 모니터링할 수 있는 대시보드 (UI는 언급되었으나, 구체적인 지표는 부족).
    *   각 스텝의 시작/종료 시간, 실행 결과, 에이전트 응답 시간, 발생한 오류 등에 대한 상세 로깅 및 메트릭.
    *   실패한 워크플로우나 지연되는 스텝에 대한 알림(alerting) 시스템.
    *   `ctx.streams` (SSE)를 활용하여 UI에 실시간 업데이트를 제공하는 것은 좋지만, 로깅 및 메트릭 시스템은 플러그인 외부에서도 접근 가능해야 합니다.
*   **버전 관리 (Versioning):**
    *   워크플로우, 도구, 지식 베이스 정의의 버전 관리에 대한 언급이 없습니다. 정의가 변경될 때 기존에 실행 중인 워크플로우는 어떻게 되는가? (예: `Workflow` 정의를 업데이트했을 때, 현재 실행 중인 인스턴스는 이전 정의를 따르는가, 새로운 정의를 따르는가?)
    *   프로덕션 환경에서는 변경 사항을 배포하고 롤백할 수 있는 명확한 전략이 필요합니다.
*   **감사 추적(Auditability):**
    *   누가 언제 어떤 워크플로우를 트리거했고, 어떤 도구를 사용했으며, 어떤 권한을 부여했는지에 대한 감사 로그가 중요합니다. `AgentToolGrant`의 `grantedBy` 필드는 시작점이지만, 더 포괄적인 감사 기능이 필요합니다.
*   **롤백 및 복구:**
    *   워크플로우가 잘못 실행되거나 예상치 못한 결과를 초래했을 때, 이를 멈추고 롤백하거나 특정 시점으로 복구할 수 있는 기능이 필요할 수 있습니다.
*   **UI/UX:**
    *   워크플로우를 정의하는 YAML은 개발자에게는 친숙하지만, 일반 사용자에게는 부담이 될 수 있습니다. 복잡한 DAG를 시각적으로 정의하고 관리할 수 있는 UI 컴포넌트(드래그 앤 드롭 등)가 장기적으로는 필요할 것입니다. `argsSchema`를 활용한 CLI 도구 인자 입력 UI도 중요합니다.
*   **데이터 유효성 검사 (Input Validation):**
    *   CLI Registry의 `argsSchema`가 선택 사항으로 되어 있는데, 모든 도구의 인자에 대한 철저한 유효성 검사가 필요합니다. 에이전트가 잘못된 인자를 전달하여 시스템에 악영향을 미치는 것을 방지해야 합니다.
*   **외부 시스템 통합:**
    *   현재 설계는 Paperclip 내부에서 완결되는 것에 초점을 맞추고 있습니다. 그러나 현실적인 업무 환경에서는 외부 API 호출, 서드파티 서비스 연동 등이 필요합니다. 워크플로우 스텝에서 이러한 외부 시스템과의 상호작용을 어떻게 안전하고 효과적으로 처리할지에 대한 고려가 필요합니다. (예: Webhook 스텝, REST API 호출 도구)
